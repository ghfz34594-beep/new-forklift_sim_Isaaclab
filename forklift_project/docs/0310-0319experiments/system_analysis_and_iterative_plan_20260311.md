# 视觉叉车系统分析与迭代实验总计划 (2026-03-11)

## 1. 目的

本文件用于把当前视觉叉车训练线的判断、目标、实验路线和记录规范统一下来，按“**假设 -> 验证 -> 修正假设 -> 再验证**”的方式持续推进，避免：

- 已经证伪的方向重复踩坑
- 不同实验之间缺乏可比性
- 只盯单个成功率指标，忽略“假成功”

本文件应作为后续一段时间内的**总实验计划与实验账本入口**。每完成一轮实验，应同步更新本文件，并将详细结果写入独立实验 md。

## 2. 当前最新结论 (2026-03-13 更新)

基于 3.1、3.2、3.3 的连贯实验，以及最新的实验 4，我们得出了非常清晰的结论：

1. **轨迹走廊 (3.1) 是有效的**：它成功规范了远场接近行为，消除了“乱走”和“远场推盘”。
2. **近场 Commit 奖励 (3.2) 容易引发作弊**：虽然它成功激发了智能体的插入意愿，但也导致了严重的 Reward Hacking。
3. **实验 3.x 的瓶颈是“死区作弊 (Reward Hacking)”**：在 3.3 中，尽管我们加重了推盘惩罚，智能体依然选择“斜着浅插推着走”。因为它发现，只要保持浅插，就能持续获得差分奖励，而这个收益大于推盘的惩罚。
4. **路线重定向至实验 4 (回归论文原生 Reward)**：鉴于差分奖励极易导致作弊，我们决定放弃复杂的势函数差分，回归原论文（绝对状态驱动 + 一票否决式惩罚）。
5. **实验 4 的初期现象（木头人）**：在修正了叉臂中心距离和动作空间后，实验 4 初期出现了“木头人”现象（停在门前刷轨迹对准分，不敢靠近）。目前正在通过微调权重（加大停滞惩罚，削弱轨迹奖励）来打破此局部最优。

### 2.1 之前路线的降级与确认

- **纯 RL (scratch)**：不可行。
- **坐标回归预训练**：已降级。`MobileNetV3 任务预训练` 与 `ResNet18 ImageNet特征+全程冻结 (RRL)` 表现一致，说明预训练不是瓶颈，RRL 是更简洁的主线。
- **动作空间**：`2 动作 approach-only` 范式已确立，剥离 `lift` 动作是正确的。

3. **`exp3 reward shaping v1` 还没有打通 `approach -> insert -> hold`**
   - 到 `iteration 348` 左右，`push_free_success_rate_total` 仅约 `0.53%`，`push_free_insert_rate_total` 仅约 `0.60%`。
   - 同时 `diag/pallet_disp_xy_mean` 仍约 `0.08m ~ 0.11m`，说明虽然出现了极少量近成功或浅插入，但距离“稳定、无推盘的插入保持成功”还很远。

4. **`loading decision / lift` 仍未开始独立打通**
   - 当前即便在 `approach-only` 主线中，也还没有形成论文那种“接近策略 + 单独决策举升”的完整 pipeline。

### 2.2 当前真正的系统状态

当前更准确的系统状态是：

- 已确认 `2 动作 approach-only` 与 `RRL (ResNet18 + ImageNet + 全程冻结)` 是当前正确主线。
- **远场接近已解决**：通过实验 3.1 的参考轨迹走廊，智能体能完美对准并开到托盘门前。
- **插入意愿已激发**：通过实验 3.2 的近场 Commit 奖励，智能体敢于把叉子插进去了（插入率可达 15%~19%）。
- **推盘惩罚已条件化**：通过实验 3.3，远场瞎撞被杜绝，但暴露了新的 Reward Hacking 漏洞。
- **当前核心卡点**：智能体在近场选择“斜着浅插推着走”，以微小的推盘惩罚换取持续的插入奖励。
- `lift / loading decision` 仍完全没打通。

### 2.3 当前最重要的负面证据

1. **特定任务预训练不是当前主要矛盾**
   - 经过实验 1 与实验 2 对比，已经可以把“要不要继续死磕坐标回归预训练”视为已回答的问题：当前主线应固定为 `RRL`。

2. **单纯加重推盘惩罚无法解决“推土机”问题**
   - 实验 3.3 证明，只要死区内依然能拿到插入奖励，智能体就会权衡利弊，选择继续推盘。必须从源头（门控）切断死区内的奖励。

3. **问题已不再是动作维度，而是 Reward 结构与诊断粒度**
   - `2 动作 approach-only` 已验证有效，不应继续把“3 动作探索噪声”当成当前主障碍。
   - 当前真正缺的是：更像论文那样的几何先验、分阶段 Reward、以及能区分 `approach / commit / insert` 的过程诊断。

## 3. 与论文《Visual-Based Forklift Learning System Enabling Zero-Shot Sim2Real Without Real-World Data》的差距

### 3.1 论文做对了什么

论文的关键成功点不是单一一个 ResNet，而是以下组合：

1. **使用标准 ImageNet 预训练 ResNet**
2. **使用双侧相机，强调叉齿与托盘开口的近场几何关系**
3. **把任务拆成 approach policy 和 loading decision**
4. **approach policy 只学驱动与转向，不把 lift 混进同一个 RL policy**
5. **在 Reward 中使用基于参考轨迹的几何先验**
   - 原文明确使用了基于 `clothoid` 近似的 reference trajectory 来计算正奖励。
6. **使用 photorealistic sim + domain randomization**
7. **reward/critic 允许使用 privileged information**

### 3.2 我们与论文的核心差距

经过实验 1 和实验 2 的修正后，我们已经在两点上向论文靠拢：

- `2 动作 approach-only`
- `RRL` 范式（ImageNet 特征 + 冻结骨干）

当前项目与论文相比，主要还差在 6 个地方：

1. **Reward 仍缺少“参考轨迹走廊”这一层全局几何先验**
   - 当前主线 Reward 仍以局部中心线误差、局部门控和插入深度为主。
   - 论文则显式利用 reference trajectory 引导 forklift 以“可插入的方式”接近托盘。

2. **相机几何仍与论文不同**
   - 当前主线是单个 `60°` 俯视相机。
   - 论文更强调从侧边观察叉齿与托盘开口关系。

3. **`approach -> commit -> insert` 还没有真正打通**
   - 当前 `exp3 v1` 说明：我们已经不只是“看不见”，而是“能靠近、能对齐一些，但不会稳定 commit 到插入”。

4. **`loading decision / lift` 仍未单独建模**
   - 论文是 `approach policy` 和 `loading decision` 两阶段。
   - 我们目前仍停留在先把 `approach` 打通的阶段。

5. **过程诊断还不够细**
   - `push-free` 指标已经补上，但还缺少能直接回答“卡在走廊、卡在 commit、还是卡在 insert”的过程指标。

6. **domain randomization 范围仍较窄**
   - 论文除了外观随机化，还对观测速度、动作执行等做扰动。
   - 这部分对后续 sim2real 仍有借鉴价值。

## 4. 系统拆解：影响因素与关系图

为了避免胡乱试超参，先把系统拆开。

### 4.1 影响最终成功率的 6 个主因素

1. **观测几何**
   - 相机数量
   - 相机高度/俯角/前向偏移
   - 是否能稳定看到叉齿和托盘开口的相对关系

2. **视觉表征精度**
   - backbone 容量
   - 分辨率
   - 数据量与数据分布
   - 监督目标是否足够密集

3. **任务拆分方式**
   - 是否将 `approach` 与 `lift/decision` 拆开
   - 是否让早期 RL 去承担过多子任务

4. **RL 优化动力学**
   - freeze/unfreeze 策略
   - backbone 学习率
   - 是否全程冻结
   - 是否延迟解冻

5. **reward / curriculum / success 判据**
   - 是否鼓励“对准后继续往前插”
   - 是否允许“推托盘式假成功”
   - 是否过早把策略训练成保守

6. **sim realism / randomization**
   - 材质、光照、颜色
   - 动作噪声、观测噪声
   - 后续 sim2real 时尤为重要

### 4.2 因果链条

当前系统更符合下面这条因果链：

`观测几何`
-> `视觉表征质量`
-> `是否能进入精对齐区域`
-> `reward 是否引导继续插入而不是推盘/停住`
-> `是否能稳定 hold`
-> `是否值得再学 lift`

这意味着：

- 如果前面的视觉表征和观测几何不过关，后面的 RL 微调和 reward 修修补补只会不断撞墙。
- 如果 reward 判据不对，即便视觉表征够了，也会被“推土机”污染。

## 5. 新的总目标与验收口径

为了避免再次被“假成功”误导，后续不再把 `success_rate_total` 作为唯一目标。

### 5.1 一级目标：先复现论文的 approach 训练逻辑

短期目标不是“端到端学完整 lift”，而是先做到：

- **纯视觉 approach policy**
- **只学 drive + steer**
- **在不推托盘的情况下完成接近 + 对齐 + 插入 + hold**

### 5.2 二级目标：再做 loading decision / lift

当一级目标稳定后，再做：

- 是否触发 lift 的判定
- lift 动作执行

### 5.3 后续统一主 KPI

建议新增并统一关注以下指标：

1. `push_free_success_rate`
   - 成功且托盘位移未超过阈值

2. `push_free_insert_rate`
   - 达到插入阈值且托盘位移未超过阈值

3. `pallet_disp_xy_mean / p95 / max`

4. `phase/frac_aligned`

5. `phase/frac_inserted`

6. `phase/hold_counter_max`

7. `diag/near_success_frac`

8. `err/lateral_near_success`

9. `err/yaw_deg_near_success`

10. `err/dist_front_mean`

11. `milestone/hit_approach`

12. 若启用 `3.x` 参考轨迹方案，增加：
    - `traj/corridor_frac`
    - `traj/commit_gate_mean`
    - `traj/d_traj_mean`

13. `phase/frac_lifted`

说明：

- `1~3` 是最终结果指标
- `4~12` 是过程诊断指标，用于判断策略到底卡在 `approach`、`commit` 还是 `insert`

### 5.4 建议的“真成功”阈值

在论文式 approach 训练阶段，建议至少满足：

- `push_free_success_rate >= 0.40` 作为中期门槛
- `push_free_success_rate >= 0.60` 作为进入下一阶段的门槛
- `pallet_disp_xy_mean <= 0.05m`
- `err/lateral_near_success <= 0.10m`
- `err/yaw_deg_near_success <= 5.0°`

说明：

- 这比当前日志里的“总成功率”更难，但也更接近真正可迁移的策略。

## 6. 迭代实验总路线 (实验 4 & 5 系列)

### 6.1 实验 4：回归论文原生绝对状态 Reward
**背景**：实验 3.x 的势函数差分奖励导致了严重的“斜插推盘”作弊。
**行动**：废弃差分奖励，回归论文的绝对状态驱动（如 $1/x$）和一票否决式惩罚。
**发现 Bug**：
1. **几何计算偏差**：发现 `dist_center` 算成了叉尖到托盘中心的距离，导致 1.2m 的巨大误差。
2. **越界逃跑**：Agent 学会了倒车逃跑以规避惩罚。
**修复**：修正几何中心计算，增加 `r_out` 越界惩罚。

### 6.2 实验 5：稀疏终局奖励 + 轻量有界 Shaping
**背景**：实验 4 修复后，Agent 变成了“木头人”，停在 1.1m 处刷轨迹对齐分。
**行动**：将无界的 $1/x$ 替换为有界的 $exp(-x/\sigma)$，防止小误差区域的数值爆炸。将终局奖励 `rg` 放大到 200 分。

### 6.3 实验 5.1：消除探索恐惧
**背景**：Agent 依然不敢靠近，怀疑是推盘惩罚（-100）导致探索瘫痪。
**行动**：将推盘惩罚降至 1.0，并极大收窄初始出生范围（喂到嘴边）。
**结果**：失败。Agent 依然停在 1.1m 处。

### 6.4 实验 5.2：修复“原地鬼畜”与轨迹奖励喧宾夺主 (关键修复)
**背景**：深入分析日志发现两个致命 Reward Hacking。
**发现 Bug**：
1. **原地鬼畜**：`rini` 惩罚只看绝对速度，Agent 学会了以 0.06m/s 原地摇晃来免罚。
2. **轨迹奖励喧宾夺主**：前进会导致偏离轨迹被扣分，不如原地不动刷分。
**修复**：
1. 将速度判定改为**向托盘的投影速度**。
2. 移除近场轨迹奖励，改用绝对姿态奖励 `r_lat` 和 `r_yaw`。
**结果**：Agent 终于被迫前进，距离缩短至 0.05m，但变成了“推土机”。

### 6.5 实验 5.3：修复距离梯度消失
**背景**：Agent 贴在门口但不插进去。
**发现 Bug**：使用了 `dist_front`（叉尖距离）并 clamp 到 0，导致碰到托盘后失去前进梯度。
**修复**：恢复使用 `dist_center`，确保插入全过程都有梯度。

### 6.6 实验 5.4：放宽 rg 阈值，打破认知障碍 (重大突破)
**背景**：Agent 变成了推土机，因为 `rg` 触发条件（误差 < 0.1m）在随机探索下物理不可达。
**行动**：将 `paper_rg_dist_thresh` 放宽到 0.4m（插进一半就算赢）。
**结果**：**巨大成功！** Agent 偶然触发大奖后瞬间开窍，`rg` 触发率飙升至 60% 以上，回合长度大幅缩短。确立了后续“基于 Checkpoint 逐步收紧阈值”的课程学习路线。

### 6.7 实验 5.5 vs 5.5b：课程退火的艺术 (步子迈大了容易扯着蛋)
**背景**：在 0.4m 阈值稳定后，我们试图直接将阈值收紧到 0.2m (Exp 5.5)。
**结果 (Exp 5.5 失败)**：`rg` 瞬间跌回 0，Agent 发现大奖太难拿，退化回了“推土机”状态（托盘位移飙升至 0.95m，偏航角恶化至 15°）。
**修正 (Exp 5.5b 平滑退火)**：
1. 将阈值退回到 **0.3m**。
2. 将推盘惩罚 `alpha_5` 从 1.0 稍微提高到 **3.0**，抑制推土机行为。
3. 重新加载 0.4m 时的 Checkpoint。
**结果 (Exp 5.5b 成功)**：Agent 完美适应了 0.3m 的新难度。
   - **成功率 (`rg`)**：稳定在 **50%** 左右（Iteration 200+ 持续保持）。
   - **姿态精度显著提升**：偏航角误差从 15° 稳步下降至 **8.9°** 左右，横向误差控制在 **0.15m**。
   - **推盘行为被有效控制**：托盘位移被压制在 **0.39m** 左右（相比于 0.2m 阈值时失控的 0.95m，`alpha_5=3.0` 的惩罚权重起到了完美的平衡作用）。
   - **结论**：确立了“平滑退火 + 动态惩罚”的 SOP，证明了在保证能拿到大奖的前提下，逐步收紧阈值可以有效逼迫 Agent 提高精度。


### 6.8 实验 5.5c：0.1m 步长在后半段依然太大 (0.3m -> 0.2m 失败)
**背景**：在 0.3m 阈值稳定后（偏航角 5°，成功率 50%+），将 `paper_rg_dist_thresh` 从 0.3m 收紧到 0.2m。
**结果 (Exp 5.5c 失败)**：训练了 **1278 个 Iteration（约 7.5 小时）**，`rg` 始终为 **0**。
**但 Agent 精度达到了惊人的水平**：
   - 偏航角误差降至 **1.6° ~ 2.5°**（手术刀级别）
   - 横向误差降至 **0.05m**
   - 托盘位移降至 **0.02 ~ 0.05m**（几乎零推盘）
   - 对准率升至 **86%**
   - `dist_front` 仅 **0.007m**（叉尖几乎贴住托盘）
**关键发现**：Agent 学会了以极高精度对准托盘口，但**拒绝往里插**。因为从未在 0.2m 下触发过 `rg`，Critic 不知道"插入"有 200 分价值。选择了"完美站在门口不进去"的局部最优。
**教训**：越接近完全插入，同样 0.1m 的跨度对精度要求呈指数级上升。**后半段步长必须缩小到 0.05m**。

### 6.9 实验 5.5d：加入 0.25m 中间台阶 (当前正在进行)
**行动**：将 `paper_rg_dist_thresh` 设为 **0.25m**（0.3m 和 0.2m 之间的中间台阶），加载 5.5c 的高精度 Checkpoint。`alpha_5` 保持 3.0 不变。
**日志**：`20260314_073134_train_exp5_5_d_smooth_curriculum_0.25.log`
**预期**：5.5c 的 Checkpoint 已有 2.5° 偏航角和 0.05m 横向误差，0.25m 相比 0.3m 只多 5cm 插入深度要求，应能较快触发成功。


### 6.10 实验 5.5e：降低推盘惩罚依然无法突破 0.25m
**背景**：在 5.5d (0.25m, alpha_5=3.0) 运行 1300+ Iteration 后 rg 依然为 0，Agent 出现探索瘫痪。
**行动**：将 `alpha_5` 从 3.0 降至 1.0，试图鼓励其往前推盘探索。
**结果 (Exp 5.5e 失败)**：Agent 确实尝试了推盘（位移上升至 0.17m），但由于 0.25m 阈值要求较深，推盘过程中托盘滑动导致无法有效缩小相对距离，最终再次放弃探索，退回"贴在门口"的局部最优。
**教训**：在 0.3m 之后，哪怕是 0.05m 的跨度都太大，降低惩罚也无法弥补物理难度。

### 6.11 实验 5.5f：终极微步退火 (0.3m -> 0.28m) (当前正在进行)
**行动**：
1. **退回起点**：加载 5.5b 的 Checkpoint (在 0.3m 阈值下有 50% 成功率的稳定状态)。
2. **微步收紧**：将 `paper_rg_dist_thresh` 设为 **0.28m**（只比 0.3m 紧 2 厘米）。
3. **保持精度**：恢复 `alpha_5 = 3.0`，防止退化为推土机。
**日志**：`20260314_081732_train_exp5_5_f_micro_step_0.28.log`
**预期**：用"切香肠"的方式，利用 5.5b 的成功记忆，引导其多往前蹭 2 厘米。
**初步结果 (破冰成功但遇瓶颈)**：在启动仅仅 3 个 Iteration 后，`rg` 就成功触发（18.75%），随后稳步上升至 **25.0%** 左右，但随后卡在这个成功率无法继续提升。这证明了"微步退火"（0.02m 步长）能够打破 0 的突破，但暴露出了新的系统性瓶颈。

### 6.13 实验 5.6：系统性 Reward 结构修正 (已完成)
**深度分析（5.5f 瓶颈所在）**：
1. **“连坐机制”（`rg` 的姿态门槛过严）**：`rg` 要求 `dist_center < THRESH & tip_y_err < 0.15 & yaw_err_deg < 10.0`。在深插时，物理碰撞会导致姿态不可避免地发生轻微变形（比如偏航角被挤到 11°）。此时距离虽然达标，但因为姿态超标，导致 `rg` 无法触发。
2. **“奖励错位”（Shaping 权重失衡）**：距离奖励 `alpha_1 = 5.0`，而姿态对齐奖励 `alpha_2 = 1.0`, `alpha_3 = 1.0`。这导致 Agent 认为“往前冲”比“摆正姿态”重要 5 倍，从而倾向于“硬挤”而不是主动调整姿态。

**行动**：
1. **放宽 `rg` 姿态门槛**：在 `env.py` 中，将 `tip_y_err < 0.15` 改为 `< 0.20`，将 `yaw_err_deg < 10.0` 改为 `< 15.0`。
2. **重塑 Shaping 权重**：在 `env_cfg.py` 中，将 `alpha_2` 和 `alpha_3` 从 1.0 提升到 5.0，与距离权重 `alpha_1` 齐平。
3. **保持推盘惩罚**：`alpha_5 = 3.0` 不变。
4. **直接挑战 0.25m**：加载 5.5b 的 0.3m checkpoint，将 `paper_rg_dist_thresh` 设为 0.25m。
**日志**：`20260314_091510_train_exp5_6_expert_reward_refactor_0.25.log`
**结果 (陷入局部最优)**：经过 800+ Iteration，`rg` 依然为 0。但是，**底层姿态指标达到了前所未有的完美状态**：偏航角误差降至 5.5°，横向误差降至 0.1m，对齐率高达 75%。推盘位移也降至极低的 0.18m。

### 6.14 深度反思：撞上物理引擎的“叹息之墙” (重大纠偏)
**警告：之前关于“局部最优”和“探索瘫痪”的分析是建立在错误归因上的伪结论！**

通过重新运行 `verify_forklift_insert_lift.py --sanity-check` 物理极限排查脚本，并查阅 2 月份的 `pallet_dragging_issue` 历史报告，我们发现了一个极其荒诞的真相：**0.25m 的阈值在物理上是绝对不可达的！**

*   **物理真相**：当前放大了 1.8 倍的托盘，内部存在阻挡柱，加上物理引擎的凸分解膨胀，导致**货叉最多只能插入 0.938 米**就会被死死卡住。
*   **数学矛盾**：`dist_center < 0.25m` 意味着要求插入深度达到 `1.2 - 0.25 = 0.95m`。
*   **结论**：`0.95m > 0.938m`。Agent 根本不是不想插，而是它已经把货叉顶到了物理引擎允许的绝对极限，却依然够不到我们设定的那条虚幻的及格线。

这是典型的“算法傲慢”——试图用修改 Reward 的方式去解决一个纯粹的物理碰撞问题。详细的自我检讨和 SOP 更新请见 `critical_reflection_physics_wall_20260314.md`。

### 6.15 实验 5.8 计划：回归物理现实的最终退火

*   **行动 3（大奖回调）**：将 `alpha_4` 从 5000 回调到正常的 **200**。
*   **行动 4（加载起点）**：加载 5.5b 的 0.3m checkpoint。


### 6.12 深度解析：为什么后半段退火必须使用"微步"？
在我们的几何定义中，`dist_center` 是**当前叉臂中心**到**目标叉臂中心（完全插入时）**的距离。叉臂总长是 1.2m。
*   **0.4m 阈值**：允许差 0.4m。意思是只要插进去 `1.2 - 0.4 = 0.8m`（插进约 2/3），算成功。
*   **0.3m 阈值**：要求插进去 `0.9m` 才能拿奖。
*   **0.28m 阈值**：要求插进去 `0.92m`。
*   **论文原版 0.1m**：要求插进去 `1.1m`（几乎要完全插到底）才能拿奖。

**为什么前半段可以大步子（0.4 -> 0.3），后半段（0.3 -> 0.2）却会摔跟头？**
这本质上是一个**"容错空间与物理碰撞的非线性关系"**问题：
1. **前半段（相对宽松）**：在 0.4m 时，叉尖刚穿过托盘底部的横梁，内部空间相对宽敞。就算叉子有点歪（比如偏航角 10°），也能硬塞进去。所以跨度 0.1m，Agent 稍微试几次就能偶然碰到大奖。
2. **后半段（极度严苛）**：当要求插入深度超过 0.9m (阈值 < 0.3m) 时，叉臂已经深深没入托盘。此时容错率呈指数级下降，**在根部哪怕只有 2° 的偏航角，在 1.2m 长的叉尖就会被放大成巨大的横向偏移**。如果它想从 0.3m 继续往里插到 0.2m，只要姿态稍微不完美，叉臂就会死死卡住托盘内壁。
3. **探索瘫痪的死锁**：一旦卡住内壁，它再往前开就会推动托盘，触发 `alpha_5=3.0` 的**推盘惩罚**。而它又因为插得不够深，拿不到 200 分的大奖。Agent 权衡利弊后觉得："往前走=扣分，停在门口=安全"，于是变成了"死活不敢插"的探索瘫痪。
4. **微步退火（如 0.28m）的意义**：0.28m 相比 0.3m **只要求多往前蹭 2 厘米**。这 2 厘米的容错空间缩小，在 Agent 现有的精度能力范围内。它偶尔一次姿态摆得稍微正一点，就能滑进这 2 厘米拿到 200 分。拿到奖之后，Critic 网络就会强化这个"更正的姿态"，从而引导 Actor 把偏航角从 5° 进一步压缩到 3°。这就是**"用切香肠的方式，一点点骗它提高精度"**的核心逻辑。

## 18. 核心底层 Bug 与物理逻辑修复总结 (Exp 2 - Exp 5)

在从实验 2 到实验 5 的推进过程中，我们发现并修复了多个属于“物理环境几何”或“底层奖励逻辑”的严重 Bug。这些 Bug 并非策略或超参问题，而是导致 Agent 陷入局部最优（如“木头人”、“推土机”、“原地鬼畜”）的根本原因。

### 18.1 几何与物理尺度 Bug
1. **`dist_center` 几何计算偏差 (在实验 4 中修复)**
   - **Bug 描述**：初期代码中，距离 `rd` 被错误地计算为“叉尖”到“托盘中心”的距离。
   - **致命后果**：叉臂长 1.2m，当叉尖碰到托盘时，代码认为距离还有 1.2m。这导致在近距离下，距离奖励的梯度极其平缓，Agent 感受不到前进的“引力”。
   - **修复方案**：严格按照物理几何，计算“叉臂几何中心”到“完全插入时的目标叉臂中心（托盘内部 0.6m）”的距离。

2. **Stage A 近场课程的出生点坐标算错，导致所谓“近场”其实出生在托盘里面 (在实验 4.1 中修复)**
   - **Bug 描述**：近场课程的 `stage1_init_x_*` 在从“叉尖到托盘前沿的目标距离”反推根部 `x_root` 时，符号和偏移量算错了。
   - **致命后果**：名义上想让叉尖出生在托盘前沿外 `0.3m ~ 0.6m`，实际却把叉车直接刷到了托盘内部或极靠前位置，导致 4.1 的课程设置和日志解读失真。
   - **修复方案**：按 `x_root = 目标距离 + pallet_front_x - fork_forward_offset` 重新推导，并把 `stage1_init_x_min/max_m` 改到真正位于托盘外的范围。

3. **`dist_front` 导致插入期梯度消失 (在实验 5.3 中修复)**
   - **Bug 描述**：为了加大近场梯度，一度将距离奖励改为基于 `dist_front`（叉尖到托盘前沿距离，且被 `clamp(min=0)` 截断）。
   - **致命后果**：当叉尖刚碰到托盘口时，`dist_front` 变为 0，距离奖励达到最大值 1.0。此后 Agent 无论插得多深，奖励都不再增加。这导致 Agent 选择停在门口“蹭蹭不进去”。
   - **修复方案**：恢复使用 `dist_center` 计算距离奖励，确保在长达 1.2m 的插入过程中，距离持续缩小，奖励持续增加。

### 18.2 奖励逻辑与状态缓存 Bug
4. **`gate_commit` 用错了距离参考点 (在实验 3.2/3.3 中修复)**
   - **Bug 描述**：近场 commit 门控和 `delta_front` 一度使用了 `stage_dist_front`，而这个量在某些配置下其实是基于车体 base 的距离，不是真实叉尖到托盘前沿的距离。
   - **致命后果**：近场门控会在“车身看起来近了、但叉尖其实没到位”时误开，导致 commit 奖励与真实插入几何错位，污染 3.2 / 3.3 的 reward 诊断。
   - **修复方案**：统一改为使用真实的 `dist_front`（叉尖到托盘前沿距离），并在 reset 时同步用真实值初始化 `_prev_dist_front`。

5. **`gate_commit` / `_prev_dist_front` 的计算时序与初始化曾出错 (在实验 3.2/3.3 中修复)**
   - **Bug 描述**：`pen_pallet_push` 在一度实现中先使用了 `gate_commit`，但 `gate_commit` 本身却在后面才计算；同时 `_prev_dist_front` 也曾在 reset 里于变量定义之前被使用，触发过 `UnboundLocalError`。
   - **致命后果**：近场 commit 奖励和条件化推盘惩罚会出现“门控值不是当前帧真实门控”的问题，甚至直接导致 reset / 训练流程报错。
   - **修复方案**：将 `gate_commit` 提前到推盘惩罚之前计算，并把 `_prev_dist_front` 的初始化移动到 `dist_front_reset` 真正算出来之后。

6. **`ra` 动作突变惩罚最初没有真实的上一帧动作缓存 (在实验 4 中修复)**
   - **Bug 描述**：虽然奖励中已经写了 `ra = -||a_t - a_{t-1}||^2`，但早期没有维护 `previous_actions` buffer。
   - **致命后果**：动作平滑惩罚要么直接失效，要么和错误/陈旧状态比较，导致我们以为“已经约束了动作抖动”，其实并没有。
   - **修复方案**：显式增加 `previous_actions` 缓存，并在 `_pre_physics_step()` 中先保存上一帧动作再写入当前动作。

7. **`rini` 停滞惩罚的“原地鬼畜”漏洞 (在实验 5.2 中修复)**
   - **Bug 描述**：初始停滞惩罚的条件是绝对速度 `torch.norm(vel[:, :2]) < 0.05`。
   - **致命后果**：Agent 极其聪明地学会了在原地“左右摇晃”或“微速倒车”，保持绝对速度在 0.06 m/s。这样既逃避了停滞惩罚，又逃避了超速惩罚，完美卡 Bug 刷取轨迹对齐分。
   - **修复方案**：将绝对速度改为**“向托盘的投影速度” (Projected Velocity)**。只有真正朝着目标前进的速度才算数，彻底封死了原地鬼畜的漏洞。

8. **越界逃跑漏洞 (在实验 4.0 中修复)**
   - **Bug 描述**：为了避免推盘惩罚和动作惩罚，Agent 发现最优解是“挂倒挡一直往后开”，逃离托盘越远越好。
   - **修复方案**：增加 `r_out` 越界惩罚（距离 > 3.0m），并加入 `out_of_bounds` 的 Episode 终止条件。

### 18.3 任务判据与 Episode 终止逻辑 Bug
9. **`approach-only` 阶段的成功判据曾错误残留 `lift` 依赖 (在实验 4 中修复)**
   - **Bug 描述**：任务已经切成 `2 动作 approach-only`，但 `hold_cond` / `success` 逻辑一度仍要求 `is_lifted` 为真。
   - **致命后果**：策略明明已经“接近 + 对准 + 插入”了，但因为没有 lift 动作，成功条件在逻辑上根本不可达，导致实验结论被系统性污染。
   - **修复方案**：在 approach-only 阶段把成功条件收敛为 `is_inserted & is_aligned`，把举升从主成功判据中拆出去。

10. **`_get_dones()` 残留旧阶段 early-stop 标志，污染了新实验的终止条件 (在实验 4 中修复)**
   - **Bug 描述**：从旧 reward 体系迁移到论文式 reward 时，`_early_stop_fly / _early_stop_stall / _early_stop_dz_stuck` 等旧版 early-stop 标志一度没有完全移除。
   - **致命后果**：新的实验（尤其是 exp4 paper-native reward）会在与当前目标无关的旧条件下被提前终止，直接扭曲 episode length、回报分布和训练信号。
   - **修复方案**：在新 reward 主线中清除这些旧 early-stop 条件，只保留与当前任务定义一致的 `tipped / success / out_of_bounds / timeout`。

### 18.4 沉淀的突破性经验
11. **`rg` 终局奖励的物理不可达阈值与“宽容篮筐”效应 (在实验 5.4 中突破)**
    - **现象**：在修复了所有漏洞后，Agent 变成了“推土机”，顶着托盘往前推，死活不插进去。
    - **根本原因**：终局奖励 `rg` 的触发条件太苛刻（`dist_center < 0.1`，要求完美插到底）。在随机探索阶段，Agent 根本不可能偶然达到这个精度，导致 Critic 网络永远不知道“插入”有 200 分的大奖。
    - **突破性经验 (动态阈值/课程学习)**：将 `rg` 的触发阈值放宽到 `0.4`（只要插进去一半就算成功）。这相当于把“篮筐”放低放大。修改后，Agent 在几代之内就偶然触发了奖励，瞬间“开窍”，成功率飙升至 60% 以上。

12. **课程退火的艺术：步子迈大了容易灾难性遗忘 (在实验 5.5b 中修正)**
    - **现象**：在 0.4m 阈值成功后，直接收紧到 0.2m，Agent 瞬间崩溃，退化回推土机。
    - **原因**：从插 2/3 到几乎插到底，对姿态精度的要求呈指数级上升，直接跳跃导致了灾难性遗忘。
    - **经验 (动态退火 SOP)**：确立了近场插入的标准操作流程：
      1. **宽容破冰**：用极宽阈值（0.4m）激活 Critic。
      2. **平滑退火**：前半段（0.4 -> 0.3）可用 0.1m 步长，**后半段（0.3 -> 0.2 -> 0.1）必须用 0.05m 步长**（因为精度要求呈指数级上升）。实验 5.5c 证明 0.3m 直跳 0.2m 会导致 1278 个 Iteration 的 rg=0。
      3. **继承记忆**：每次收紧必须加载上一阶段稳定的 Checkpoint。
      4. **动态惩罚**：在收紧阈值的同时，缓慢增加推盘惩罚（如 1.0 -> 3.0），逼迫其走向精细控制。
      5. **失败的 Checkpoint 也有价值**：即使 rg=0，Agent 可能学到了极高的精度（如 5.5c 的 2.5° 偏航角），在下一阶段可以复用。

13. **轨迹奖励在近场的“喧宾夺主”效应 (在实验 5.2 中修正)**
14. **视觉表征的“伪瓶颈”与 RRL 范式的确立 (Exp 1 vs Exp 2)**
   - **现象**：在早期实验中，我们花费了大量精力进行 MobileNetV3 的坐标回归预训练，试图让 CNN 直接输出精确的 `(x, y, yaw)`。但无论怎么训练，Y 轴误差始终降不下去（约 16cm），且最终 RL 成功率卡死在 16.5%。
   - **根本原因（认知偏差）**：我们错误地认为“如果 CNN 不能输出精确的物理坐标，RL 就学不会”。但实际上，论文采用的是 **RRL (ResNet as Representation for RL)** 范式，即直接使用标准的 ImageNet 预训练特征，并且**全程冻结骨干网络**。
   - **突破性经验**：在实验 2 中，我们直接换用冻结的 ResNet18 (ImageNet 权重)，不加任何物理坐标预训练，结果发现其表现与辛辛苦苦预训练的 MobileNetV3 完全一致！这证明了**当前的瓶颈根本不在视觉感知端，而在于 RL 的探索策略和 Reward 塑造**。将“理解视觉特征的物理含义”的任务交给下游的 MLP (Actor/Critic) 去学习，是更简洁、更鲁棒的路线。

15. **动作空间的降维打击 (Exp 1 & Exp 4)**
   - **现象**：最初我们让 Agent 同时控制 `drive` (驱动), `steer` (转向) 和 `lift` (举升) 三个动作。结果 Agent 在接近托盘时，经常因为胡乱举升导致叉车重心不稳、翻车，或者叉尖高度不对而撞击托盘。
   - **根本原因**：将“导航接近”和“举升操作”混在一个 RL 策略中，极大地增加了探索空间的维度和噪声。在还没有学会“走到门口”之前，就让它乱动叉子，是典型的“还没学会走就想跑”。
   - **突破性经验**：在实验 1 和实验 4 中，我们坚决贯彻了论文的思路，将动作空间从 3 维降到 2 维 (`approach-only` 策略，只控制 `drive` 和 `steer`)，强制 `lift` 关节固定为 0。这极大地收敛了探索空间，使得 Agent 能够专心学习平面上的对齐和插入。这证明了**在复杂机器人任务中，合理的动作空间解耦比端到端“大力出奇迹”更重要**。

16. **势函数差分奖励的“薅羊毛”陷阱 (Exp 3.x)**
   - **现象**：在实验 3.x 中，我们尝试使用复杂的势函数差分（如 `dist_front` 的下降量）来奖励插入。结果 Agent 学会了“斜着浅插推着走”：它发现只要保持在托盘边缘摩擦，就能不断产生微小的“距离下降”，从而源源不断地薅取差分奖励，而这笔收益竟然超过了推盘的惩罚。
   - **根本原因**：差分奖励（基于状态变化量）在连续控制中极易被利用。Agent 可以通过高频的微小震荡或卡在某个临界点，将单次任务变成“无限提款机”。
   - **突破性经验**：在实验 4 中，我们果断废弃了差分奖励，回归论文原生的**绝对状态驱动奖励**（如 $exp(-dist)$）。绝对状态奖励只看“你现在离目标有多近”，而不是“你刚才靠近了多少”，从数学上彻底封死了“原地摩擦刷分”的漏洞。

### 18.5 其他实现级脏 Bug（不改变方向，但会污染实验）
- **`rini` 缩进错误**：曾导致 exp4 中停滞惩罚实现位置不对，属于典型的“不是理论问题，是代码写歪了”的脏 bug。
- **实验 5.2 日志 key 未同步 (`r_traj_*` -> `r_lat/r_yaw`)**：一度导致训练启动后直接报 `NameError`，说明 reward 重构后日志字段也必须同步重构。
- **这类 bug 的共同教训**：以后凡是大改 reward 结构，都必须同时检查 `reward 计算`、`buffer 初始化`、`log key`、`done 条件`、`reset 初始化` 这五个面，不能只改公式本身。

### 6.16 实验 5.8 总结与 5.9 计划：核弹级奖励 + Early Stopping (能力嫁接)
**Exp 5.8 结果**：在 0.28m 阈值和 200 分奖励下，成功率突破 40%，但 Agent 出现了“战略性摆烂”——为了拿到奖励，姿态变差（偏航 13°），推盘增加（0.6m）。这证明了 200 分的奖励下，学习过于缓慢，导致姿态在探索中退化。

**Exp 5.9 计划 (基于用户敏锐的直觉)**：
*   **核心假设**：利用 **Exp 5.6 的“完美姿态” Checkpoint** 作为起点，给予 **5000 分的核弹级奖励**。巨大的梯度会让成功率在极短的迭代内（20-50代）飙升。在这个短暂的窗口期，神经网络权重来不及大幅漂移，从而保留 5.6 的完美姿态。只要在“黄金交叉点”果断 **Early Stopping**，就能把高成功率和完美姿态嫁接在一起。
*   **行动**：
    *   保持 `paper_rg_dist_thresh = 0.28`。
    *   将 `alpha_4` 提升至 **5000.0**。
    *   加载 Exp 5.6 的 Checkpoint。

## 7. 物理环境重大修正：视觉盲区与渲染降级 (2026-03-15)
在开启 Exp 7 之前，我们进行了一次极其关键的物理层面的排查，发现了导致过去几天模型频繁出现“推土机”行为的底层原因，并进行了重大修正。详情见：[物理环境重大修正：视觉盲区消除与渲染性能权衡分析](./critical_fix_camera_blind_spot_and_rendering_20260315.md)。

**核心修正内容：**
1. **消除物理盲区**：将相机俯仰角从 60° 调整至 75°，确保货叉根部和完全插入的托盘始终在视野内。
2. **解决渲染降级导致的“隐身”问题**：在 Headless 训练模式下，由于材质和光照的丢失，黑色货叉与深色地面融为一体。我们通过强制将货叉 USD 材质覆盖为**亮红色**，极大地提升了目标对比度，让神经网络不再“瞎眼”。

由于视觉输入发生了根本性改变，旧的特征提取器全部失效，后续的实验（Exp 7.x）将基于此全新环境**从头开始训练 (Scratch)**。
