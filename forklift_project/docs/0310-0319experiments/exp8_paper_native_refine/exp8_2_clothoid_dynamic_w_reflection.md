# Exp 8.2 实验结果与深度反思：Clothoid 曲线的成功与轨迹几何 Bug 的发现

**日期**：2026-03-19
**实验分支**：`exp/vision_cnn/exp8_paper_native_refine` (Run: `exp8_2_clothoid_dynamic_w`)

## 1. 实验背景与核心配置

在之前的 Exp 8.1 中，我们通过释放绝对状态奖励的引力（去掉防爆常数 $c$）和降低惩罚权重，成功打破了 Agent 在 1.9m 处"匀速绕圈"的局部最优，使其敢于冲锋到托盘前 0.35m。然而，由于使用的是 Bézier 曲线，其在近端的曲率变化不够平滑，无法提供精确的偏航角引导，导致 Agent 在最后半米陷入"无效摩擦"，最终退缩。

为了彻底解决"末端对准"问题，我们在 Exp 8.2 中严格对齐了论文的核心思想，进行了以下重构：

1. **引入 Clothoid 曲线近似**：使用三次 Hermite 样条替换 Bézier 曲线，模拟 Clothoid 曲线曲率线性变化的特性，并**强制终点切线严格等于 0°（与托盘平行）**。
2. **动态权重增强末端引导**：在 `dist_front < 0.5m` 时，将偏航角对齐奖励 $r_{c\psi}$ 的权重放大 3 倍。
3. **引入 $L_{bound}$ 替代部分 $r_a$**：在奖励中加入 `r_bound = -||a||`，惩罚动作绝对值，防止方向盘打死。
4. **严格对齐惩罚阈值**：确认 $r_p$（推盘惩罚）是基于速度的二值惩罚；确认 $r_v$（超速惩罚）的存在。

## 2. 实验结果现象：从"坚守阵地"到"全面崩溃"

实验运行至 765 代，呈现出先扬后抑的戏剧性变化：

### 阶段一：完美的姿态控制与坚守 (Iter 0 ~ 500)
- **现象**：Clothoid 曲线的威力显现。Agent 停在距离托盘 **0.5m ~ 0.6m** 处，偏航角误差从 Exp 8.1 的 23° 大幅下降并稳定在 **13° ~ 20°** 之间。
- **行为**：推盘位移极小（~0.2m），推盘惩罚极低。
- **结论**：Clothoid 曲线和动态权重成功"锚定"了 Agent，使其在长达 500 代的时间里顶住了"久攻不下"的压力。

### 阶段二：防线失守，全面崩溃 (Iter 500 ~ 765)
- **数据**：`dist_front` 退回到了 **1.07m**，偏航角飙升至 **63.1°**，推盘位移暴增至 **1.14m**，插入率跌至 **4.6%**。

## 3. 深度诊断：推翻"视觉盲区"假设，发现轨迹几何 Bug

### 3.1 视觉特征验证：ResNet34 足以区分不同距离

我们的初步诊断是"冻结的 ResNet34 在 0.5m 处看不懂几何语义"。

**然而，通过 `analyze_resnet34_similarity.py` 的定量分析，这个假设被推翻了：**
- 相邻 0.1m 图像的平均特征余弦相似度约为 **0.9882**（不是 1.0）。
- 特征相似度随距离差增大而**逐步下降**，说明 ResNet34 确实编码了距离变化。
- 从 +0.5m 到 -0.8m 的整个序列中，特征变化是**连续且可区分的**。

**结论：视觉特征在 0.5m 处是充分的。** Agent 看到了清晰的托盘，ResNet34 也能区分不同的距离和角度。

（详细分析结果见 `exp8_2_resnet34_similarity_analysis/` 子文件夹）

### 3.2 真正的罪魁祸首：轨迹跟踪了错误的对象

既然视觉没问题，为什么 Agent 的偏航角死活卡在 20° 左右？

我们编写了 `check_traj_tangent.py` 来验证轨迹的数学属性，发现了一个**极其致命的几何 Bug**：

**轨迹跟踪的是车体中心（`root_pos`），而不是叉臂中心（`fork_center`）！**

论文里写得清清楚楚：
> "$r_d$ and $r_{cd}$ are the distances from the **center of the forks** to the pallet and clothoid curve"

但我们的代码中：
```python
# _build_reference_trajectory 中
p0 = self.robot.data.root_pos_w[env_ids, :2]  # 车体中心！

# _query_reference_trajectory 中
root_pos = self.robot.data.root_pos_w[:, :2]   # 还是车体中心！
```

**后果**：叉尖在车体中心前方 **1.87 米**。当叉尖到达托盘前沿（准备插入）时，车体中心还在 1.87m 处。因为 `traj_pre_dist_m = 1.2m`（直线段只有 1.2m），此时车体中心依然处于**轨迹的曲线段上**。

经脚本验证，在车体中心 = 1.87m 处，轨迹的切线角度是 **-17.35 度**。

**破案了：Agent 的 20° 偏航角是"正确答案"（根据错误的轨迹）！**
它在完美地执行我们给它的错误指令——轨迹告诉它"在这个位置你应该偏 20 度"，它照做了，拿满了 $r_{c\psi}$ 分，但因为车头是歪的，永远也插不进托盘。

## 4. 修复方案与 Exp 8.3

修复极其简单，只需要改两行代码：

1. `_build_reference_trajectory` 中：起点从 `root_pos` 改为 `fork_center`
2. `_query_reference_trajectory` 中：查询点从 `root_pos` 改为 `fork_center`

这样，轨迹就会引导"叉臂中心"平滑地驶入托盘。当叉尖到达托盘前沿时，叉臂中心刚好在直线段上，切线角度完美等于 0 度。

**Exp 8.3 已基于此修复启动训练（Run: `exp8_3_fork_center_traj`）。**

## 5. 反思：为什么这个 Bug 隐藏了这么久？

1. **直觉陷阱**：我们自然地认为"叉车的轨迹"就是"叉车中心的轨迹"，而忘了论文中明确要求的是"叉臂中心的轨迹"。
2. **症状误导**：Agent 在 0.5m 处偏航角卡在 20° 且无法插入，这个症状非常容易被误诊为"视觉不够精确"或"奖励梯度不够强"。实际上，它是在完美地完成一个错误的任务。
3. **铁律再次验证**："传感器先于算法"这条铁律不仅适用于物理传感器，也适用于**数学参考信号的正确性**。在奖励函数中，参考轨迹就是 Agent 的"传感器"——如果传感器给出了错误的方向，再强大的算法也无法弥补。
