# MobileNetV3 视觉预训练 + RL 微调实验结果与下一轮计划 (2026-03-10)

## 1. 实验目标

验证“**专家数据集位姿预训练 + RL 微调**”这条路线，是否能突破纯视觉 RL 从零训练（`exp/vision_cnn/cnn_pretrain`）约 **20%** 成功率的平台。

---

## 2. 本轮实验配置

- **分支**：`exp/vision_cnn/rl_finetune`
- **初始化权重**：`outputs/vision_pretrain/best_backbone.pt`
- **训练日志**：`logs/20260310_225914_train_s1.0ze_finetune.log`
- **训练规模**：`num_envs=256`，`max_iterations=2000`
- **视觉输入**：`64x64`，60 度俯视相机
- **初始设想**：前 500 次更新冻结 Backbone，再解冻联合微调

## 3. 关键结果

### 3.1 与纯 RL Baseline 对比

| 指标 | 纯 RL Baseline | 预训练 + 微调 | 结论 |
|---|---:|---:|---|
| `episode/success_rate_total` 最终值 | 20.29% | **25.98%** | 提升 **+5.69 个百分点** |
| `episode/success_rate_total` 峰值 | 约 20.3% | **26.32%** | 明显提升 |
| `diag/near_success_frac` 最终值 | 23.05% | **39.06%** | 提升 **+16.01 个百分点** |
| `err/lateral_near_success` | 0.22m ~ 0.30m | **0.231m** | 最终没有本质改善 |
| `err/yaw_deg_near_success` | 9° ~ 11° | **8.02°** | 略好，但仍在阈值边缘 |
| `diag/pallet_disp_xy_mean` | 1.7m ~ 2.1m | **0.661m** | 推盘现象显著减轻 |

### 3.2 本轮训练最终状态

- **最终总成功率**：`25.98%`
- **最终 EMA 成功率**：`30.04%`
- **最后 100 iter 平均成功率**：`25.83%`
- **最终接近成功率**：`39.06%`
- **最后 100 iter 平均接近成功率**：`40.82%`
- **最终深插入比例**：`28.12%`
- **最终接近成功横向误差**：`0.231m`
- **最后 100 iter 平均横向误差**：`0.246m`
- **最终接近成功偏航误差**：`8.02°`
- **最后 100 iter 平均偏航误差**：`8.90°`
- **`phase/frac_lifted` 最大值**：`0.0`

### 3.3 中间过程中的关键现象

本轮实验最重要的发现，不是“最终 26% 比 20% 高”，而是下面这组现象：

| Iteration | `success_rate_total` | `near_success_frac` | `err/yaw_deg_near_success` | `err/lateral_near_success` |
|---|---:|---:|---:|---:|
| 50 | 4.54% | 15.23% | **3.80°** | **0.117m** |
| 100 | 6.35% | 19.92% | **4.54°** | **0.109m** |
| 500 | 15.60% | 17.19% | 9.33° | 0.228m |
| 1000 | 22.73% | 19.53% | 7.97° | 0.199m |
| 1500 | 25.94% | 41.02% | 10.09° | 0.260m |
| 1999 | 25.98% | 39.06% | 8.02° | 0.231m |

也就是说：

1. **训练早期**，策略已经拥有了非常好的对齐精度。
2. **训练后期**，成功率继续上升，但对齐精度反而退化了。
3. 这说明预训练特征是有效的，但在 RL 微调过程中没有被稳定保住。

## 4. 假设 -> 验证 -> 修正假设

### 假设 H1：视觉预训练可以突破 20% 平台

- **验证结果**：成立，但只成立了一半。
- **证据**：
  - 最终成功率从 20.29% 提升到 25.98%
  - 接近成功率从 23.05% 提升到 39.06%
  - 托盘平均位移显著下降，说明“推盘”明显减轻
- **修正后的结论**：
  - 预训练确实让视觉表征更强了
  - 但它没有自动解决“最后一小段精对齐 + 保持 + 举升”的问题

### 假设 H2：当前的首要瓶颈是 `64x64` 分辨率太低

- **验证结果**：暂时**不能**下这个结论。
- **证据**：
  - Iteration 50/100 时，横向误差已经到 `0.117m / 0.109m`
  - Iteration 50/100 时，偏航误差已经到 `3.80° / 4.54°`
  - 这些数值已经明显优于纯 RL Baseline 后期
- **修正后的结论**：
  - `64x64 + 预训练` 至少在训练早期已经能提供足够好的几何线索
  - 当前更大的问题是：**RL 微调把这份精度“学丢了”**

### 假设 H3：冻结 500 次更新可以保护预训练特征

- **验证结果**：当前实验没有真正验证到这个假设。
- **关键发现**：
  - 日志中出现了 `Vision backbone is now trainable`
  - 它出现在 **Iteration 7** 附近，而不是我们原本以为的“500 个训练迭代之后”
- **原因**：
  - 当前 `freeze_backbone_updates=500` 的计数口径，不是 PPO 的外层 iteration，而是更细粒度的内部更新调用次数
- **修正后的结论**：
  - 这次实验**不是**一个“冻结 500 iter 再解冻”的干净实验
  - 因此，下一轮最高优先级不是先升分辨率，而是先把**冻结/解冻策略做对**

### 假设 H4：一旦接近和插入改善，举升会自然学会

- **验证结果**：不成立。
- **证据**：
  - `phase/frac_lifted` 全程最大值为 `0.0`
- **修正后的结论**：
  - 举升阶段不是自动“顺带学会”的
  - 它要么需要更稳定的插入保持，要么需要单独的课程/奖励设计来打通

## 5. 系统因素分析

把整个系统拆开后，可以更清楚地看出关系链：

1. **视觉表征质量**
   - 由数据集、预训练任务、分辨率、Backbone 决定
   - 直接影响叉车能否看出托盘的精确位置和偏航

2. **RL 微调策略**
   - 由冻结策略、学习率、是否端到端更新决定
   - 直接影响预训练特征是被保住、轻微适配，还是被 PPO 梯度破坏

3. **控制策略学习**
   - 决定“知道托盘在哪”之后，能不能把车开到正确位置
   - 体现在 `near_success_frac`、`deep_insert_frac` 上

4. **插入后保持与举升**
   - 决定能不能从“插进去一点”变成“稳定保持并举升完成”
   - 体现在 `phase/frac_lifted`、`lift_*` 指标上

5. **课程/奖励/成功判据**
   - 决定策略会把优化重点放在哪个阶段
   - 如果插入后保持和举升奖励不够强，策略就会学成“靠近并顶进去一点”的局部最优

因此当前的主链条是：

**预训练视觉表征有效**  
→ **早期精对齐很好**  
→ **RL 微调没有稳定保住精度**  
→ **后期变成“能靠近、能插一部分，但保持不住，也不会举升”**  
→ **成功率停在 26% 左右**

## 6. 明确结论

### 结论 1
**“视觉预训练 + RL 微调”这条路线是正确的，值得继续投入。**

原因是它同时改善了：

- 总成功率
- 接近成功率
- 深插入比例
- 托盘位移（减少推盘）

### 结论 2
**当前最优先要解决的问题，不是先升分辨率，而是先把微调策略做对。**

原因是：

- 早期指标已经证明预训练特征本身是够用的
- 真正的问题是后续 PPO 更新把精对齐能力破坏掉了

### 结论 3
**这次实验没有干净验证“冻结 500 iter”的假设，因此下一轮必须先做单因素纠错实验。**

### 结论 4
**举升阶段目前完全没打通，必须被单独当成一个子问题来看。**

### 结论 5：视觉验收发现“推托盘”局部最优陷阱 (Reward Hacking)
通过对最佳模型 (`model_1400.pt`) 的双视角录屏分析，发现了一个严重的异常行为：**叉车在未对准的情况下，会直接撞击并推着托盘往前走；即使偶然挤进去一部分，也会一直死踩油门往前推，而不会执行起升（Lift）动作。**
- **原因分析**：当前分支的奖励函数与 `master` 分支（90%成功率的专家策略）完全一致。推托盘惩罚 (`pen_pallet_push`) 的配置为：权重 1.0，死区 5cm，插入深度 > 30% 时惩罚归零。专家策略拥有精确的低维坐标，能精准对齐，根本不触发 5cm 的死区；而视觉策略“看”得模糊，学不会复杂的“倒车微调”，只能选择“硬顶”。
- **奖励漏洞被放大**：1.0 的惩罚权重敌不过往前开带来的 `r_pot`（接近目标的潜力奖励）。一旦瞎猫碰死耗子挤进托盘 30%，惩罚彻底消失。此时 Critic 发现“一直往前开”是一个零风险、稳收益的高价值状态，导致 Actor 陷入“无脑推土机”的局部最优。
- **启示**：针对精确低维状态调优的奖励函数，在面对高维模糊的视觉输入时，其鲁棒性被击穿。这也直接促成了我们在下一轮计划中加入“实验 4：奖励函数修复”。

## 7. 下一轮实验计划

下面按照“假设 -> 单因素验证 -> 决策”的方式推进。

### 实验 0：先修复实验基础设施

**目标**：保证下一轮实验是可复现、可比较的。

需要做的事情：

1. 修正 `freeze_backbone_updates` 的计数口径，让它按**外层训练 iteration**工作，而不是按内部 update 调用次数工作。
2. 给日志增加明确打印：
   - 第几次 iteration 冻结
   - 第几次 iteration 解冻
3. 修正预训练脚本输出路径的相对路径问题，避免权重再次落到仓库外部目录。

**完成标准**：

- 能在日志中清晰看到“冻结到第 N 个 iteration，解冻发生在第 N+1 个 iteration”

---

### 实验 1：全程冻结 Backbone（最高优先级，对照实验）

**假设 H1**：如果把 Backbone 全程冻结，只训练视觉后端 MLP 和 PPO 控制头，那么早期的高精度视觉特征可以被稳定保住，成功率会超过本次 26%。

**只改一个因素**：

- `freeze_backbone=True`
- 冻结整个 2000 iter，不解冻

**保持不变**：

- 数据集不变
- 预训练权重不变
- `64x64` 分辨率不变
- 奖励、课程、环境不变

**验证指标**：

- `err/lateral_near_success <= 0.18m`
- `err/yaw_deg_near_success <= 7°`
- `episode/success_rate_total >= 30%`
- `phase/frac_lifted` 是否开始大于 0

**决策规则**：

- 如果成功率和精度同时提升，说明主问题是“微调破坏预训练特征”
- 如果精度保住了但成功率仍不高，说明还需要继续优化控制或课程

---

### 实验 2：真正“冻结前 500 iter，再解冻”

**假设 H2**：如果 Backbone 在前 500 个训练 iteration 真的被冻结，Actor/Critic Head 会先适应特征，然后再做小幅端到端微调，最终效果会优于“全程冻结”和本次实验。

**只改一个因素**：

- 修复冻结口径后，设置前 500 iter 冻结，之后解冻

**验证指标**：

- Iter 100 时保持早期对齐精度
- Iter 500 后成功率继续上升，但对齐误差不能像本次一样明显恶化
- 最终 `success_rate_total >= 30%`

**决策规则**：

- 如果优于实验 1，后续就沿“延迟解冻 / 更慢解冻 / 更低 backbone lr”方向继续优化
- 如果不优于实验 1，说明当前任务下更适合长期冻结 Backbone

---

### 实验 3：只有在实验 1/2 失败后，再升到 `128x128`

**假设 H3**：如果冻结策略修正之后，最终精度仍然稳定卡在 `y > 0.18m` 或 `yaw > 7°`，那么 `64x64` 的信息上限才是下一层主要瓶颈。

**只改一个因素**：

- 相机分辨率从 `64x64` 升到 `128x128`

**要配套做的事情**：

1. 重新采集专家数据集
2. 重新进行视觉预训练
3. 在修正过冻结策略的 RL 流程中重新微调

**验证指标**：

- 预训练阶段：`Y MAE <= 0.10m`，`Yaw MAE <= 2.5°`
- RL 阶段：`success_rate_total >= 35%`

---

### 实验 4：奖励函数修复（推盘惩罚与举升引导）

**启动条件**：当视觉特征（冻结或微调）被证明有效，但 Agent 仍然表现出“一直推着托盘走”、“插进去不举升”的局部最优行为时。

**假设 H4**：当前的奖励函数（尤其是 `pen_pallet_push` 的门控和死区）在视觉策略下存在漏洞，导致 Agent 陷入了“死踩油门不举升”的局部最优。

**验证动作**：
1. **恢复强推盘惩罚**：将 `k_pallet_push_pen` 从 1.0 恢复到 3.0，或缩小死区，观察是否能抑制“盲目冲撞”。
2. **取消插入豁免**：修改 `push_gate`，即使插进去了，只要没起升，推托盘依然要受轻微惩罚。
3. **增加超时惩罚/举升引导**：如果插入深度达到了 80% 以上，但迟迟不执行 Lift 动作，就开始扣分，逼迫它从“推土机”状态切换到“起升”状态。

**验证指标**：
- 观察录屏，推托盘现象是否消失。
- `phase/frac_lifted > 0`
- `milestone/lift_10cm_frac > 0`

### 实验 5：举升阶段专项实验（课程学习）

**启动条件**：如果实验 4 修复了奖励漏洞，但 Agent 依然学不会举升。

**假设 H5**：当前不是“看不见 lift”，而是从初始状态走到举升状态的轨迹太长，探索不到。

**只改一个因素**：
- 课程学习改成更靠近插入完成态的出生分布。

**验证指标**：
- 最终能否出现稳定举升成功。

## 9. 当前建议的执行顺序

按优先级，下一步不要直接跳到更重的 `128x128` 全链路重跑，而是按下面顺序推进：

1. **修复冻结口径和日志打印**
2. **做“全程冻结 Backbone”对照实验**
3. **做“真正冻结前 500 iter 再解冻”实验**
4. **只有在 1~3 仍然失败时，再升到 `128x128`**
5. **最后再单独打通举升阶段**

这条路线最省实验成本，也最符合“先修正错误假设，再做更重实验”的原则。
