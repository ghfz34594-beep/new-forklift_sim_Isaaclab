# 环境物理极限与成功判定逻辑修复总结 (2026-03-16)

## 1. 背景与发现
在观察 Exp 7.1 的训练日志时，我们发现一个矛盾现象：模型已经获得了较高的 `rg` 奖励（意味着货叉已经深入托盘），但日志中的 `phase/frac_inserted`（插入成功率）始终为 0.0000。

经排查，发现 `env.py` 中存在一个**严重的历史遗留 Bug**：
```python
# 旧代码
insert_norm = torch.clamp(insert_depth / (self.cfg.pallet_depth_m + 1e-6), 0.0, 1.0)
is_inserted = insert_norm > 0.8  # 硬编码了 80% 的插入阈值
```

## 2. 物理可达性重新验证
针对“80% 插入深度是否物理可达”的问题，我们停止了训练，并运行了最新版本的物理极限验证脚本 (`verify_forklift_insert_lift.py --sanity-check`)。

**验证脚本自身修复**：
在运行前，我们修复了验证脚本中几处会污染结论的陈旧逻辑：
1. 成功判定对齐了当前 `env.py` 的逻辑（仅要求插入+对齐，不再强制要求举升）。
2. `max_insert` 的计算从简单的 X 坐标相减，改为了与 `env.py` 一致的沿托盘局部坐标系投影计算。
3. 修复了 `stage_1_mode` 下举升动作被禁用导致的误报。

**物理极限实测结果**：
*   **当前托盘总深度**：`2.16m` (1.2m * 1.8x 缩放)
*   **物理最大插入深度 (B1 测试)**：**`0.9091m`**。当货叉插入到这个深度时，会被托盘内部的 `convexDecomposition` 碰撞体死死卡住，无法继续前进。
*   **80% 深度要求**：`2.16m * 0.8 = 1.728m`。
*   **结论**：`1.728m > 0.9091m`。**80% 的插入深度在物理上绝对不可达。**

## 3. 修复与调整
1. **统一成功阈值**：
   将 `env.py` 中的硬编码 `0.8` 替换为读取配置文件中的 `insert_fraction`。
   ```python
   # 修复后代码
   is_inserted = insert_norm >= self.cfg.insert_fraction
   ```
2. **确认当前配置的合理性**：
   目前 `env_cfg.py` 中配置的 `insert_fraction = 0.40`。
   `2.16m * 0.40 = 0.864m`。
   `0.864m < 0.9091m`（物理极限）。
   这证明当前的 **40% 阈值是物理可达的**，且留有约 4.5 厘米的安全余量。

## 4. 影响评估与下一步
*   **对之前训练的影响**：由于 `is_inserted` 变量不仅用于日志统计，还直接参与了 `hold_counter` 的累积和 `success` 终止条件的判定，这个硬编码 Bug 导致模型在之前的训练中**永远无法触发真正的 Success 终止**。
*   **下一步行动**：环境的物理逻辑和判定逻辑现已彻底理顺。接下来应基于修复后的环境，重新启动 Exp 7.1 的训练。此时日志中的 `frac_inserted` 将能真实反映模型是否达到了 40% 的物理可达插入标准。
