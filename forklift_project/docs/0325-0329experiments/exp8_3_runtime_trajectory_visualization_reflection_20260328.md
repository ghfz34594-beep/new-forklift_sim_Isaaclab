# Exp8.3 参考轨迹可视化缺位的反思与检讨

日期：2026-03-28

## 1. 这次要反思什么

这次 runtime 轨迹可视化做完之后，暴露出一个很扎眼的问题：

**我们在设计和反复修改参考轨迹相关实验时，长期没有把“真实 runtime 生成的轨迹到底长什么样”作为第一优先级验证项。**

这不是一个小疏漏，而是研究流程上的系统性缺口。

## 2. 为什么这是一个严重失误

因为参考轨迹在这条实验主线里，不是边角料，而是核心机制之一。

我们前面围绕它做了很多事情：

- 修了 `fork_center` 作为轨迹起点
- 调了 `traj_pre_dist_m`
- 比较了 `g2 / g3`
- 讨论了 `r_cd / r_cpsi`
- 做了很多 reward / curriculum / success 逻辑实验

但在相当长一段时间里，我们验证的主要是：

- 数学几何是否一致
- reward 指标是否上升
- success / insert / dirty 是否变化
- rollout 视频看起来是否“像在做任务”

而没有优先做下面这个最直接的问题：

**“训练时 env 真正生成的参考轨迹，入口段是不是合理？”**

这说明我们的验证链条顺序是反的。

## 3. 这次失误的具体原因是什么

我认为至少有 4 个原因。

### 3.1 过早把“轨迹存在”当成了“轨迹合理”

之前我们知道系统里有：

- `_build_reference_trajectory()`
- `traj_pre_dist_m`
- `r_cd / r_cpsi`

于是很容易在心理上默认：

“既然轨迹都建出来了，而且几何预检也通过了，那 runtime 轨迹大概率就是合理的。”

这是错误的。

**“有轨迹”不等于“轨迹入口几何没问题”。**

### 3.2 过早把 rollout 结果当成轨迹机制本身的验证

之前看视频时，我们更容易关注：

- 会不会靠近
- 会不会插
- 会不会推盘

但 rollout 视频本质上是“策略行为”的结果。

它不能自动告诉我们：

- 参考轨迹本身有没有把 steering 变成必要动作
- 起点和 `p_pre` 的关系是不是反了

也就是说，我们把“行为结果”当成了“机制验证”的替代品。

### 3.3 对“训练 shaping”和“推理利用”的区别认识不够彻底

这次重新看代码后才更明确地意识到：

- 轨迹在当前系统里主要通过 reward shaping 起作用
- actor 并没有直接拿到 signed steering 几何量

在这种架构下，轨迹是否合理，反而更需要通过可视化去确认。  
因为它不是一个显式控制器，不能保证“只要轨迹在，就会自动教会 steering”。

之前没有把这一点提到足够高的优先级，是一个认知上的缺口。

### 3.4 研究流程里缺少“runtime geometry visualization gate”

更本质地说，是流程没有立规矩。

我们有：

- smoke test
- short train
- unified eval
- zero-steer grid

但没有一条硬规则说：

**任何涉及参考轨迹几何的改动，都必须先通过 runtime top-down 可视化检查，再允许进训练。**

这导致实验可以在“几何关系未被直观看懂”的情况下继续向前推进。

## 4. 这次教训是什么

这次最大的教训不是“以后多看几张图”，而是：

**几何机制类实验，不能只靠数值指标和最终行为回推，必须先直视 runtime 几何本体。**

对这条主线来说，以后至少要把下面这些当成标准前置检查：

1. 运行时参考轨迹俯视图
2. 托盘坐标系视图
3. `s_start / s_pre / s_goal / delta_s` 直接标注
4. 代表性 case 的 rollout overlay

如果这些没有先通过，就不应该急着开大批训练。

## 5. 我对这次问题的检讨

如果要直说，我这次的问题不是“不会做可视化”，而是：

**没有足够早地把“先把轨迹本身看清楚”排到最前面。**

我在分析时更快地沿着：

- reward 是否有效
- clean insert 是否上升
- success 是否打开
- 是否该继续扫超参数

这些方向往前走了。

这在工程推进上看起来很积极，但在这个问题上，实际上绕开了一个更基础的验证点。

这使得我们：

- 花了不少实验成本
- 才在后面通过 steering diagnostics 和 runtime 轨迹图，回头发现入口几何本身就有问题

这说明我在研究流程上有一个需要修正的习惯：

**当一个机制是“几何定义 + reward shaping + PPO 学习”这种复合结构时，必须先验证几何定义本身，再讨论学习是否成功。**

## 6. 后面怎么避免重犯

我建议把下面这条规则写成以后这类实验的固定流程：

### 凡是改了轨迹相关配置，先做 4 步

1. 画 runtime 参考轨迹
2. 画托盘坐标系下的 `s/y` 关系
3. 检查 `s_start < s_pre < s_goal` 是否成立
4. 只在几何关系正确时，才进入 reward / training 实验

### 凡是说“策略学会 steering 了”，必须补 2 个证据

1. `normal > zero-steer`
2. runtime grid / overlay 证明 steering gap 来自几何利用，而不是偶然训练噪声

## 7. 一句最坦白的总结

这次最大的反思是：

**我们之前一直在研究“怎么让 agent 学会一条轨迹”，却没有先把“这条轨迹在 runtime 里到底是不是以正确方式出现”看清楚。**

这是一个流程级错误，不只是一个小疏忽。

这次补上的 runtime 可视化，不只是帮我们看图，而是把后续实验顺序重新摆正了。
