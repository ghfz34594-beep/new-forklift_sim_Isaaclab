# 256x256 分辨率纯 RL Scratch Baseline 实验结论 (2026-03-11)

## 1. 实验背景与目标

在发现 64x64 分辨率下，视觉策略容易陷入“推托盘”的局部最优后，我们假设核心瓶颈在于 **64x64 分辨率太低，导致特征模糊，Agent 无法进行精细的几何对齐**。

为了验证这个假设，我们将相机分辨率升级到了 **256x256**。在进行 256x256 的预训练和 Finetune 之前，我们需要先跑一个**纯 RL 从零开始（Scratch Baseline）**的对照实验，以确认：**如果不做预训练，纯靠 PPO 算法直接端到端学习 256x256 的高维图像输入，效率和效果如何？**

## 2. 实验配置

- **分支**：`exp/vision_cnn/scratch_baseline`
- **训练日志**：`logs/20260311_132447_train_scratch_baseline_256x256_env64.log`
- **训练规模**：`num_envs=64`（由于 256x256 显存消耗极大，从 256 降至 64 避免 OOM）
- **视觉输入**：`256x256`，60 度俯视相机
- **Backbone**：MobileNetV3-Small，**未经过预训练**（随机初始化或 ImageNet 初始化）
- **冻结策略**：不冻结，全程端到端联合训练

## 3. 实验过程与结果

该实验运行至 **Learning iteration 111** 时被手动终止（耗时约 1 小时，ETA 约 16 小时）。

### 3.1 核心指标表现（截止 Iteration 111）

- **`episode/success_rate_total`**：**0.0000** (全程为 0)
- **`phase/frac_inserted`**：**0.0000** (全程为 0)
- **`phase/frac_lifted`**：**0.0000** (全程为 0)
- **`phase/frac_aligned`**：最高达到约 **0.50**
- **`milestone/hit_fine_align`**：极少出现，最高为 **0.0156** (约 1/64)
- **`milestone/hit_precise_align`**：极少出现，最高为 **0.0156** (约 1/64)

### 3.2 现象分析

1. **完全无法学会插入**：在 111 代（约 45 万步）的训练后，Agent 连一次成功的插入都没有做到。
2. **极低的探索效率**：虽然 `frac_aligned` 能达到 0.5，说明 Agent 学会了“靠近”托盘，但 `hit_fine_align` 和 `hit_precise_align` 几乎为 0。这意味着它只是粗略地把车开到了托盘附近，但完全无法完成对准插孔所需的精细姿态调整。
3. **计算成本极其高昂**：在 256x256 分辨率下，即使只有 64 个环境，单次 Iteration 耗时也长达 25~35 秒。如果让它跑完 2000 代，需要超过 16 个小时。

## 4. 结论

**结论：纯 RL 从零开始（Scratch）学习 256x256 高分辨率视觉特征，在当前任务中是完全不可行的。**

- **特征学习难度过大**：高分辨率带来了指数级增长的输入维度（256x256x3），纯靠 RL 稀疏且高方差的奖励信号（尤其是初期只有 `r_pot` 这种粗糙的距离奖励），根本无法在合理时间内驱动 CNN 提取出有用的几何特征（如叉齿与插孔的相对位姿）。
- **证明了预训练的绝对必要性**：这个 Baseline 完美地充当了“反面教材”。它证明了我们之前决定走**“专家数据监督预训练 + RL Finetune”**的路线是完全正确的。如果不做预训练，即使给了高分辨率的清晰图像，Agent 也是“睁眼瞎”。

## 5. 下一步行动

基于此结论，我们已终止该 Scratch Baseline，并立即启动了 **256x256 预训练特征 + Finetune** 的核心验证实验（日志：`20260311_142441_train_finetune_256x256_env64.log`），以观察预训练能否打破这一僵局。
