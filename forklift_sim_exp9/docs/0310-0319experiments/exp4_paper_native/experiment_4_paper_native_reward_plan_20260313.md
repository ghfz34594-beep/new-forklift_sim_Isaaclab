# 实验 4：回归论文原生 Reward 体系 (Paper Native Reward)

## 1. 背景与动机
在实验 3.x 系列中，我们尝试使用“势函数差分 (Potential-based Shaping)”和复杂的条件门控来引导智能体。虽然成功解决了远场接近问题，但陷入了严重的 Reward Hacking（智能体通过斜着浅插推着走来刷取差分奖励）。

重新审视原论文公式后发现，论文使用的是**绝对状态驱动**的奖励，且推盘惩罚是**基于速度的一票否决式**。这避免了差分奖励带来的“必须不断移动才能得分”的副作用。

## 2. 核心假设 (H4)
放弃复杂的势函数差分，直接采用论文公式 6 和 7 的绝对状态奖励和速度惩罚，能够让智能体在靠近托盘后“安静地待着”并完成插入，而不会变成“推土机”。

## 3. Reward 公式重构 (对应论文 Eq.6 & Eq.7)

### 3.1 正向奖励 (Positive Reward R+)
```text
R+ = α1 * (1 / (rd + eps)) + α2 * (1 / (rcd + eps)) + α3 * (1 / (rcψ + eps)) + α4 * rg + r_lift
```
- `rd`: **叉臂几何中心**到托盘中心的距离（**重大修正**：原先使用叉尖到托盘表面的距离，导致智能体停在托盘前不插入。使用中心距离能迫使智能体完全插入，使两个中心重合）。
- `rcd`: 叉车到参考轨迹的距离 (`d_traj`)
- `rcψ`: 叉车偏航角与轨迹切线的误差 (`yaw_traj_err_rad`)
- `rg`: 到达托盘时的特殊奖励 (当 `rd < 0.1m` 且姿态对准时给 1，否则 0)
- `r_lift`: 举升奖励（**论文补丁**：原论文中 RL 仅负责 2 维 approach，不负责举升。为了实现端到端，我们在 `rg` 触发时，根据举升高度给予额外奖励）。

### 3.2 负向惩罚 (Penalty Reward R-)
```text
R- = α5 * rp + α6 * rv + α7 * ra + α8 * rini
```
- `rp`: 推盘惩罚。如果托盘速度 `||v_p|| > 0.01`，则 `rp = -1`，否则为 0。（**重大修正**：为防止 Reward Hacking，将 `α5` 的权重从 1.0 放大到 100.0，以抗衡靠近时极大的 `1/rd` 奖励）。
- `rv`: 超速惩罚。如果叉车速度 `||v|| > 0.07`，则 `rv = -(||v|| - 0.07)^2`。
- `ra`: 动作突变惩罚。`ra = -||a - a_old||^2`。
- `rini`: 初始停滞惩罚。如果 `||v|| < 0.05` 且 `rd > 0.3`，则 `rini = -1`。

## 4. 实施计划与重大修正 (2026-03-13 更新)
在初次运行实验 4 时，我们发现了原论文中的几个“隐藏技巧”，并进行了相应的代码修正：
1. **动作空间降维**：在 `env_cfg.py` 中将 `action_space` 从 3 改为 2（仅保留 drive 和 steer），因为论文中的 RL 智能体不负责举升。举升动作通过自动补 0 处理，或者通过后续的补丁奖励引导。
2. **重写距离 `rd`**：在 `env.py` 中新增 `_compute_fork_center` 方法，计算叉臂几何中心，并将其到托盘中心的距离作为 `rd`。
3. **平衡惩罚权重**：在 `env_cfg.py` 中将 `alpha_5` (推盘惩罚权重) 放大至 100.0。
4. **修改成功条件**：在 `env.py` 的 `hold_cond` 中移除 `is_lifted` 条件，只要插入且对准即算作成功（与论文一致）。

## 5. 实验记录与动态调整 (2026-03-13)

### 5.1 首次运行现象：从“推土机”变成“木头人”
在应用了上述修正并开启训练后（跑到约 250 代），我们观察到了全新的 Reward Hacking 行为：
*   **现象**：智能体不再推盘（`rp` 几乎为 0），并且完美对准了参考轨迹（偏航误差降至 7°，横向误差降至 0.47m）。但是，它**停在距离托盘约 0.92 米处不再前进**，导致所有环境都在原地耗到超时（Episode length 达到 1079）。
*   **原因分析**：典型的公式设计导致的局部最优解。轨迹对准奖励（`r_traj_yaw` 和 `r_traj_d`）使用 `1/x` 形式，在误差较小时数值爆炸，每步能提供约 19 分。而靠近托盘的奖励（`r_dist`，权重 1.0）仅提供 0.4 分，初始停滞惩罚（`rini`，权重 1.0）仅扣 1 分。智能体为了保住高额的轨迹分数，并避免触发极高的推盘惩罚（100分），选择停在原地“刷分”。

### 5.2 权重微调：打破舒适区
为了打破这种局部最优，我们中断了训练，并对 `env_cfg.py` 中的权重进行了微调：
*   **加大停滞惩罚（逼它动）**：将 `alpha_8` (rini) 从 1.0 提高到 **20.0**。停在原地就狠狠扣分。
*   **削弱轨迹奖励的统治力（防刷分）**：将 `alpha_2` 和 `alpha_3` (轨迹距离和角度) 从 1.0 降到 **0.5**。
*   **提高靠近奖励（给甜头）**：将 `alpha_1` (距离托盘) 从 1.0 提高到 **5.0**。

### 5.3 第二次运行现象：从“木头人”变成“逃兵”
应用上述权重后，智能体确实不敢停在原地了（`rini` 惩罚变为 0），但它发明了新的作弊方式：**挂倒挡往后开**。
*   **现象**：`err/dist_front_mean` 从初始的 0.5 米一路飙升到 5.0 米以上，且存活时间接近满分。
*   **原因分析**：智能体发现，只要顺着轨迹往后倒车，既能避免停滞惩罚，又能绝对安全地避开推盘惩罚（100分），同时还能持续白嫖轨迹对齐的奖励。
*   **解决方案（堵漏洞）**：
    1. **增加距离越界终止条件**：在 `_get_dones` 中增加 `out_of_bounds` 条件，当 `dist_center > 3.0` 米时直接判定失败，斩断其无限倒车刷分的可能。
    2. **增加越界惩罚**：在 Reward 中增加一项，一旦越界给予一次性大额惩罚（如 -50），让“逃跑”的代价大于“靠近可能撞盘”的代价。
